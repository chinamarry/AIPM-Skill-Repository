## Description

本skill用于从Axure导出的原型HTML文件夹深度扫描所有页面字段、筛选条件、新增弹窗、约束条件说明，生成结构化的PRD Markdown文档，供AI编程使用。

**适用场景**：
- 从Axure原型HTML生成PRD文档
- 深度扫描所有字段信息（包括下拉框、模糊搜索框、二级级联框、动态面板）
- 提取约束条件、使用场景、页面说明等备注文字
- 生成页面交互关系树和关键交互路径

## Execution Flow（8步）

| Step | 任务 | 执行内容 | 输出 | 用户确认 |
|------|------|---------|------|---------|
| **Step 1** | 扫描所有HTML页面文件 | 列出全部页面文件名、路径、数量，识别页面类型（列表页/详情页/弹窗页） | 页面清单表 | ❌ 不需要 |
| **Step 2** | 深度扫描筛选条件区域 | 根据坐标位置（data-top=107/105）识别筛选条件区域，提取所有控件ID、控件类型、坐标位置 | 筛选条件清单（含待确认项） | ✅ **需要确认Predictive search对应字段名** |

## Key Methodologies（关键方法论）

### 1. 坐标位置分析法

根据Axure导出HTML中的控件坐标位置判断字段所属区域：

| 区域类型 | data-top | data-height | 判断规则 |
|---------|----------|-------------|---------|
| **统计区域** | 89 | 88px | 关键指标统计区域（列表页顶部） |
| **筛选条件区域** | 107 或 105 | 32px | 筛选条件区域控件 |
| **表格列** | 200+ | 44px | 表格数据区域控件 |
| **弹窗区域** | 300+ | 44px | 弹窗内控件 |

**执行方法**：
```bash
# 搜索筛选条件区域控件（data-top=107/105）
grep -n "data-top.*107" *.html
grep -n "data-top.*105" *.html
```

---

### 2. Predictive search控件识别规则

Predictive search控件在Axure导出HTML中没有明确标签，需要特殊处理：

| 规则 | 说明 |
|------|------|
| **无明确标签** | Predictive search控件在HTML中没有字段名标签，只有控件ID |
| **坐标排序** | 根据data-left坐标从小到大排序，确定从左到右顺序 |
| **用户确认** | 需要用户确认每个Predictive search对应的字段名 |

**执行方法**：
1. 搜索所有Predictive search控件：
   ```bash
   grep -n "Predictive search" *.html
   ```
2. 提取控件ID和坐标位置：
   ```bash
   grep -n "data-label=\"Predictive search\"" *.html
   ```
3. 根据data-left坐标排序，列出从左到右顺序
4. **中断执行，向用户确认每个Predictive search对应的字段名**

---

### 3. 约束条件深度扫描

Axure原型中通常包含约束条件、使用场景、页面说明等备注文字，需要深度扫描：

| 搜索关键词 | 提取内容 |
|-----------|---------|
| "约束条件说明" | 字段校验规则、必填规则、联动规则 |
| "解释说明" | 字段详细说明 |
| "必填项说明" | 必填字段列表 |
| "限定X汉字以内" | 字段长度校验规则 |
| "不可修改" | 只读字段说明 |
| "自动填充" | 联动规则说明 |

**执行方法**：
```bash
# 搜索约束条件说明
grep -n "约束条件说明" *.html
grep -n "解释说明" *.html
grep -n "必填项说明" *.html
grep -n "限定.*汉字" *.html
```

### 4. 下拉框枚举值提取

提取所有下拉框的option value列表：

**执行方法**：
```bash
# 搜索下拉框option value
grep -n "<option.*value=" *.html
```

**输出格式**：
| 下拉框字段名 | 枚举值列表 |
|-------------|-----------|
| 商机状态 | 全部商机状态 / 正常 / 挂起 / 失单 / 赢单 |
| 拜访形式 | 线下拜访 / 线上会议 |

---

### 5. 二级级联联动规则识别

识别下拉框之间的二级级联联动关系：

| 一级字段 | 二级字段 | 联动规则 |
|---------|---------|---------|
| 归属区域（省） | 归属区域（市） | 选择省后，市下拉框动态加载对应城市 |
| 客户类别（类型） | 客户类别（子类型） | 选择类型后，子类型下拉框动态加载对应子类型 |

**执行方法**：
1. 搜索data.js文件中的OnSelectionChange事件
2. 提取动态面板切换逻辑
3. 整理联动规则表

### 9. Tab页结构完整识别（新增2026-04-14）

| 规则 | 说明 |
|------|------|
| **搜索panel_state结构** | 使用`grep -n "panel_state.*data-label"`搜索所有Tab页 |

**执行方法**：
```bash
# 搜索所有Tab页结构

```

**输出格式**：
| Tab序号 | data-label | style | 显示状态 |
|--------|-----------|-------|---------|
| state0 | 基本信息 | style="" | 默认显示 |
| state1 | 状态变更日志 | style="visibility: hidden;" | 隐藏 |
| state2 | 关联售前项目 | style="visibility: hidden;" | 隐藏 |
| state3 | 关联交付项目 | style="visibility: hidden;" | 隐藏 |

### 10. 导航层级关系图生成规则（新增2026-04-14）

| 规则 | 说明 |
|------|------|
| **默认显示Tab必须列出** | 基本信息（state0）是默认显示的Tab，必须在导航层级关系图中列出 |
| **Tab内编辑弹窗归属** | 【Tab内编辑】弹窗必须放在对应的【Tab】下，不能直接放在详情页下 |
| **详情页命名规则** | 详情页节点名称使用"XXX详情页"，不包含Tab名称 |
| **Tab节点命名规则** | Tab节点名称使用"【Tab】XXX"，明确标注为Tab页 |
| **列表页按钮完整扫描** | 列表页的"新增"和"修改"按钮通常共用一个弹窗，需要合并为一个节点"【弹窗】新增/修改XXX" |

**错误示例1**（Tab结构遗漏）：
```
│   │   └── 📄 【L1→L2】线索商机详情页           ← 商机管理→"详情"触发
│   │       ├── 📄 【Tab内编辑】商机编辑弹窗      ← 错误：直接放在详情页下，遗漏【Tab】基本信息
│   │       ├── 📄 【Tab】状态变更日志           ← 详情内Tab切换
│   │       └── 📄 【Tab】关联售前项目           ← 详情内Tab切换
```

**错误示例2**（列表页按钮遗漏）：
```
│   └── 📄 【L1】线索商机管理                  ← 一级入口
│       ├── 📄 【弹窗】新增商机                ← 错误：只列出"新增"，遗漏"修改"
│       └── 📄 【L1→L2】线索商机详情页           ← 商机管理→"详情"触发
```

**正确示例**：
```
│   └── 📄 【L1】线索商机管理                  ← 一级入口
│       ├── 📄 【弹窗】新增/修改商机            ← 正确：合并"新增"和"修改"为一个节点
│       └── 📄 【L1→L2】线索商机详情页           ← 商机管理→"详情"触发
│           ├── 📄 【Tab】基本信息              ← 默认显示的Tab（state0）
│           │   └── 📄 【Tab内编辑】商机编辑弹窗  ← 正确：放在基本信息Tab下
│           ├── 📄 【Tab】状态变更日志           ← 详情内Tab切换（state1）
│           ├── 📄 【Tab】关联售前项目           ← 详情内Tab切换（state2）
│           └── 📄 【Tab】关联交付项目           ← 详情内Tab切换（state3）
```

**检查方法**：
- 搜索详情页HTML中的所有`panel_state`结构
- 确认state0（默认显示Tab）是否在导航层级关系图中列出
- 确认Tab内编辑弹窗是否放在对应的Tab节点下

#### 25. 自我一致性校验规则（新增2026-04-14）

| 规则 | 说明 |
|------|------|
| **表格列字段排列顺序校验** | 在生成PRD文档后，必须验证表格列字段的排列顺序是否与HTML中的排序顺序完全一致 |
| **坐标位置比对** | 提取HTML中表格列字段的data-left坐标位置，按照从小到大排序，与PRD文档中的表格列字段排列顺序进行比对 |
| **字段编号一致性** | PRD文档中的表格列字段编号（如VT1、VT2、VT3...）必须与HTML中的从左到右顺序一致 |
| **校验时机** | 在Step 8生成PRD文档后，必须进行自我一致性校验，确保表格列字段排列顺序与HTML一致 |

**执行方法**：
```bash
# Step 1：提取HTML中表格列字段的data-left坐标位置
grep -n "data-top.*25.*data-height.*44" *.html | grep "Table header" | sort -t':' -k3 -n

# Step 2：按照data-left坐标从小到大排序，列出字段名称
# 例如：data-left="249" → VT1，data-left="368" → VT2，data-left="570" → VT3...

# Step 3：比对PRD文档中的表格列字段排列顺序
# 如果不一致，修正PRD文档中的表格列字段排列顺序

# Step 4：验证所有页面的表格列字段排列顺序
# 对每个页面都执行上述校验流程
```

**校验流程**：
1. 提取HTML中表格列字段的data-left坐标位置
2. 按照data-left坐标从小到大排序，列出字段名称和编号（VT1、VT2、VT3...）
3. 比对PRD文档中的表格列字段排列顺序
4. 如果不一致，修正PRD文档中的表格列字段排列顺序
5. 对所有页面都执行上述校验流程

**常见不一致案例**：
| 页面 | HTML中的顺序 | PRD中的顺序（错误） | 修正后的顺序 |
|------|-------------|-------------------|-------------|
| 客户拜访管理 | 拜访日期 → 拜访客户 → 拜访对象 → 干系人分类 → 拜访形式 → 拜访目的 | 拜访客户 → 拜访目的 → 拜访日期 → 拜访对象 | 拜访日期 → 拜访客户 → 拜访对象 → 干系人分类 → 拜访形式 → 拜访目的 |
| 客户干系人管理 | 干系人姓名 → 干系人分类 → 干系人角色 → 手机号码 → 办公电话 → 电子邮箱 | 干系人姓名 → 手机号码 → 干系人分类 → 干系人角色 | 干系人姓名 → 干系人分类 → 干系人角色 → 手机号码 → 办公电话 → 电子邮箱 |

**重要提示**：
- 表格列字段的排列顺序必须与HTML中的排序顺序完全一致，不能有任何偏差
- 字段编号必须按照从左到右的顺序编号（VT1、VT2、VT3...），不能根据逻辑顺序编号
- 自我一致性校验是PRD文档质量保证的关键步骤，必须严格执行

#### 26. 删除操作设计差异识别规则（新增2026-04-14）

| 规则 | 说明 |
|------|------|
| **删除操作设计差异** | 不同页面的删除操作设计存在差异，需要识别哪些页面有删除操作，哪些页面没有删除操作 |
| **有删除操作的页面** | 客户详情-干系人Tab、客户详情-拜访记录Tab、客户干系人管理（全局列表）、客户拜访管理（全局列表） |
| **没有删除操作的页面** | 客户库管理（操作列只包含"修改 / 详情"）、线索商机管理（操作列只包含"修改 / 详情 / 更改状态"） |
| **业务规则与原型设计矛盾识别** | 如果业务规则中提到了删除操作的限制条件（如"客户删除前，需校验该客户是否关联有干系人或拜访记录，如存在则禁止删除并提示"），但原型HTML中没有删除按钮，需要向用户确认设计意图 |

**执行方法**：
```bash
# Step 1：搜索所有页面的操作列内容
grep -n "操作.*修改.*详情.*删除" *.html

# Step 2：识别有删除操作的页面
grep -n "修改 / 删除" *.html

# Step 3：识别没有删除操作的页面
grep -n "修改 / 详情" *.html

# Step 4：搜索业务规则中的删除相关内容
grep -n "删除.*校验|禁止删除" *.md
```

**删除操作设计差异案例**：
| 页面类型 | 操作列内容 | 是否有删除操作 | 说明 |
|---------|-----------|---------------|------|
| **客户库管理** | 修改 / 详情 | ❌ 无删除操作 | 原型HTML中没有删除按钮，业务规则BR-005提到删除限制条件，存在矛盾 |
| **线索商机管理** | 修改 / 详情 / 更改状态 | ❌ 无删除操作 | 原型HTML中没有删除按钮，数据权限说明："只有创建者本人可以执行'修改和更改状态'的操作" |
| **客户详情-干系人Tab** | 修改 / 删除 | ✅ 有删除操作 | 原型HTML中有删除按钮，需要补充删除确认弹窗 |
| **客户详情-拜访记录Tab** | 修改 / 删除 | ✅ 有删除操作 | 原型HTML中有删除按钮，需要补充删除确认弹窗 |
| **客户干系人管理（全局列表）** | 修改 / 删除 | ✅ 有删除操作 | 原型HTML中有删除按钮，需要补充删除确认弹窗 |
| **客户拜访管理（全局列表）** | 修改 / 删除 | ✅ 有删除操作 | 原型HTML中有删除按钮，需要补充删除确认弹窗 |

**业务规则与原型设计矛盾处理流程**：
1. **识别矛盾**：如果业务规则中提到了删除操作的限制条件，但原型HTML中没有删除按钮
2. **向用户确认设计意图**：
   - 选项A：客户和商机确实不允许删除，应该删除业务规则中的删除限制条件
   - 选项B：允许删除，应该在原型HTML中添加删除按钮，并在PRD文档中补充删除操作说明
3. **根据用户确认结果更新PRD文档**：
   - 如果选择选项A，删除业务规则中的删除限制条件，并在PRD文档中说明"客户和商机不允许删除"
   - 如果选择选项B，在PRD文档中补充删除操作说明（删除按钮位置、删除确认弹窗、删除限制条件）

**重要提示**：
- 删除操作设计差异是原型设计中的常见问题，需要仔细识别
- 业务规则与原型设计矛盾需要向用户确认，不能自行判断
- 删除操作通常需要删除确认弹窗，避免误删

## Common Errors & Solutions（常见错误与解决方案）

### 错误1：Predictive search控件识别错误

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| Predictive search控件没有明确标签 | Axure导出HTML只显示控件ID | 根据坐标位置确定顺序，用户确认字段名 |

### 错误2：筛选条件区域遗漏

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 只搜索带明确标签的字段 | 扫描策略不够全面 | 扫描筛选条件区域所有控件ID和坐标位置 |

### 错误3：表格列遗漏

## Example Usage

**用户**：从Axure原型HTML生成PRD文档

**执行流程**：
1. Step 1：扫描所有HTML页面文件 → 输出页面清单表
2. Step 2：深度扫描筛选条件区域 → 输出筛选条件清单（含待确认项）
3. **中断执行**：向用户确认Predictive search控件对应的字段名

5. Step 4：深度扫描新增/编辑弹窗 → 输出弹窗字段清单

7. **中断执行**：向用户确认约束条件是否完整
8. Step 6：深度扫描下拉框枚举值 → 输出枚举值清单
9. Step 7：整合UI设计规范 → 输出UI规范映射表
10. Step 8：生成PRD Markdown文档 → 输出PRD文档
11. **中断执行**：向用户确认PRD文档是否满足需求
